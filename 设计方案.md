# 跨平台无线音频分发系统技术方案（需求对齐优化版）
**文档版本**：v4.1  
**最后更新**：2026年5月24日  
**适用范围**：Windows 7+ / Windows 10+ → Android 4.4+，固定 WiFi 局域网内多安卓设备音频播放、声道分配与 DLNA 兼容  
**核心目标**：安卓声音端可被 Windows 发送端和安卓配置端发现、识别、命名、配置；Windows 端可作为系统音频输出设备；声卡模式优先低延迟和同步，DLNA 模式支持单机和多机声道配置但需做同步风险控制。

## 1. 产品目标与边界
### 1.1 三端角色
| 端 | 核心职责 | 必须满足 |
|----|----------|----------|
| Android 声音端 | 接收并播放局域网音频；发布设备身份；支持电脑声卡模式和 DLNA 模式 | Android 4.4+、后台接收、固定识别码、别名可改、可配置声道 |
| Windows 发送端 | 扫描声音端；作为系统音频输出；采集、编码、分发音频；配置声道 | 启动扫描、显示信号/识别码/别名、勾选使用、左/右/2.1/5.1/7.1 配置、虚拟声卡驱动 |
| Android 配置端 | 扫描声音端并远程配置工作模式和声道 | 支持电脑声卡模式、DLNA 单机模式、DLNA 多机模式、状态查看 |

### 1.2 模式定位
| 模式 | 技术路径 | 是否推荐多机同步 | 定位 |
|------|----------|------------------|------|
| 电脑声卡模式 | Windows 虚拟声卡/WASAPI → RTP/UDP → Android 播放 | 推荐 | 主模式，低延迟、高音质、可控同步 |
| DLNA 单机模式 | Windows/DLNA 控制器 → 单台 Android DMR | 推荐 | 兼容电视、播放器、普通投屏场景 |
| DLNA 多机模式 | 同一音轨同时推送到多台 Android DMR，并按设备声道播放 | 有条件支持 | 满足需求，但必须增加同源音轨、统一起播、延迟测量和手动校准 |

### 1.3 关键边界
1. **低延迟和稳定同步以电脑声卡模式为主**，DLNA 协议本身不提供严格多设备同步。
2. **DLNA 多机模式可以做**，但必须把所有设备使用同一音源、同一播放进度、同一声道切片，并提供延迟补偿；不能承诺所有手机天然无延迟。
3. **Android 4.4 必须支持声音端基础播放**，但高精度同步、低延迟能力按设备分级启用。
4. **Windows 完美兼容系统音频输出**需要虚拟声卡驱动，WASAPI 环回只能作为第一阶段验证方案。

## 2. 性能目标
| 指标 | 目标值 | 条件 | 说明 |
|------|--------|------|------|
| 声卡模式端到端延迟 | 40-80ms | Android 8+ / 5GHz WiFi | 游戏、视频优先 |
| 声卡模式端到端延迟 | 80-150ms | Android 4.4-7.1 / 2.4GHz WiFi | 兼容优先 |
| 声卡模式设备间偏差 | 5-20ms | Android 8+ / 同型号或高性能设备 | 通过 PTP/缓冲/ASRC 控制 |
| 声卡模式设备间偏差 | 20-50ms | Android 4.4-7.1 | 普通同步，允许降级 |
| DLNA 单机延迟 | 200-1000ms | 标准 DLNA 播放 | 不作为低延迟路径 |
| DLNA 多机偏差 | 目标 20-80ms，依设备而定 | 同一 WiFi、同一音源、完成校准 | 需要明确提示用户校准 |
| 音频质量 | 输入支持 48kHz/16-bit 或 48kHz/24-bit，Opus 64-256kbps | 声卡模式 | Opus 是有损编码，不写成“24-bit Opus” |
| 并发设备 | 4-8 台优先保证，最多 12 台尝试 | 家用路由器 | 超过 8 台建议降低码率或合并声道 |

## 3. Android 声音端设计
### 3.1 设备身份
**模块 ID**：AND-DEVICE-ID-001

1. 首次启动生成 `UUIDv4`，保存到应用私有存储，作为主识别码。
2. 别名由 Windows 发送端或 Android 配置端设置，别名变更不改变主识别码。
3. 辅助信息包含设备型号、Android 版本、应用版本、IP、WiFi 信息、可选 MAC/ANDROID_ID。
4. 应用重装后主 UUID 可能变化，发送端可用“设备型号 + 辅助标识 + 用户确认”尝试恢复历史别名和配置。

### 3.2 声卡模式接收
**模块 ID**：AND-RTP-RECV-001

1. 接收 Windows 端 RTP/UDP 音频包。
2. 支持 Opus 解码，优先 48kHz，按设备能力输出 16-bit PCM 或 float PCM。
3. 每个声音端只播放被分配的声道：左、右、低音炮、中置、环绕等。
4. 实现抖动缓冲、丢包隐藏、序列号重排、时间戳播放。
5. Android 4.4 使用 `AudioTrack`/OpenSL ES 低延迟路径；Android 8+ 可使用 AAudio/Oboe。

### 3.3 DLNA 模式接收
**模块 ID**：AND-DLNA-DMR-001

1. 声音端作为 DLNA Digital Media Renderer 暴露到局域网。
2. 支持 DLNA 单机播放：一台设备完整播放音轨。
3. 支持 DLNA 多机播放：多台设备接收同一音源，各自只输出配置声道。
4. 多机模式必须使用发送端统一控制起播，禁止每台设备各自选择不同媒体或不同进度。
5. 多机模式需要每台设备保存 `dlnaDelayMs` 补偿值，支持自动测量和手动微调。
6. UI 必须提示：DLNA 多机同步受设备解码缓冲和系统播放器影响，若左右声道不同步，建议切换电脑声卡模式。

### 3.4 后台运行与状态上报
1. 使用前台服务保持接收稳定，避免 Android 后台杀进程。
2. 上报在线状态、RSSI、丢包率、缓冲水位、当前模式、声道、延迟补偿。
3. 接收端断线后自动重连，配置保持不丢失。

## 4. Windows 发送端设计
### 4.1 发现与设备列表
**模块 ID**：WIN-DISCOVERY-001

1. 点击“开始扫描”后扫描局域网声音端。
2. 建议统一使用 `mDNS + UDP 广播`：mDNS 用于标准服务发现，UDP 广播用于兼容老设备和多网卡环境。
3. 设备列表显示：勾选状态、信号状态、识别码、别名、IP、Android 版本、当前模式、声道、延迟补偿。
4. 支持手动选择网卡，避免 VPN、虚拟网卡、Docker 网卡干扰。
5. 别名修改后通过控制通道写入声音端，并在 Windows 本地持久化。

### 4.2 系统音频输出兼容
**模块 ID**：WIN-VIRTUAL-AUDIO-001

1. Phase 1：WASAPI Loopback 捕获，用于快速验证局域网播放和同步。
2. Phase 2：虚拟声卡驱动，作为系统默认音频输出设备。
3. 虚拟声卡需要支持 2.0、2.1、5.1、7.1，优先 48kHz/16-bit，扩展支持 48kHz/24-bit。
4. Windows 7+ 兼容目标意味着驱动签名、安装器、卸载恢复、默认设备切换都必须纳入产品化任务。
5. 对“独占模式音频”和游戏低延迟场景，必须走虚拟声卡路径，不能只依赖 WASAPI 环回。

### 4.3 声道配置
**模块 ID**：WIN-CHANNEL-CFG-001

1. 基础模式：勾选声音端后快速配置为左声道或右声道。
2. 高级模式：支持 2.1、5.1、7.1 拖拽式声道分配。
3. 每个设备可配置一个或多个声道，但默认一个设备一个主要声道，降低混音复杂度。
4. 支持测试音：点击左声道、右声道、中置、低音炮等，验证实际设备输出。
5. 支持保存多套场景：卧室立体声、客厅 5.1、全屋播放等。

### 4.4 音频发送与同步
1. 声卡模式使用 RTP/UDP 单播到每台设备，避免 WiFi 组播不稳定。
2. 每个音频块携带全局播放时间戳、声道标识、编码格式、帧序号。
3. RTCP 或控制通道收集丢包、抖动、缓冲、时钟漂移。
4. 发送端按设备反馈动态调整码率、FEC、目标缓冲和延迟补偿。

## 5. Android 配置端设计
### 5.1 扫描与管理
**模块 ID**：AND-CFG-SCAN-001

1. 扫描局域网内所有声音端，显示识别码、别名、IP、信号、当前模式、在线状态。
2. 支持批量选择设备并统一配置模式。
3. 配置端和 Windows 端使用同一套控制协议，避免配置结果不一致。

### 5.2 模式配置
**模块 ID**：AND-CFG-MODE-001

| 配置模式 | 可配置项 | 注意事项 |
|----------|----------|----------|
| 电脑声卡模式 | 声道、缓冲、低延迟/高质量、延迟补偿 | 推荐多设备同步使用 |
| DLNA 单机模式 | 设备别名、播放质量、完整音轨输出 | 单设备兼容播放 |
| DLNA 多机模式 | 左/右/低音炮/环绕声道、统一起播、延迟补偿 | 必须同源音轨，必须校准 |

### 5.3 DLNA 多机配置规则
1. 多机模式下所有设备必须加入同一个 `dlnaGroupId`。
2. 一个组内必须使用同一个媒体 URL、同一个起播命令、同一个基准进度。
3. 左声道和右声道不能分别推不同来源音轨，只能从同一音轨拆声道后输出。
4. 提供“左右同步测试”，播放短脉冲或节拍音，用户可微调每台设备延迟。
5. 当检测到偏差长期大于 80ms，提示切换到电脑声卡模式。

## 6. 统一协议与端口
### 6.1 端口建议
| 端口 | 协议 | 用途 |
|------|------|------|
| 5004 | UDP | RTP 音频数据 |
| 5005 | UDP | 时间同步 / PTP-like 同步 |
| 5006 | TCP | HTTP/JSON 控制 API |
| 5353 | UDP | mDNS 服务发现 |
| 1900 | UDP | SSDP / DLNA 发现 |
| 8200 | TCP | DLNA 媒体服务或代理 |

> 当前项目文档和代码已经使用 5004/5005/5006/mDNS，建议不要再切到 17080 系列，除非同步修改所有端。

### 6.2 发现协议
1. 声音端发布 mDNS 服务：`_soundplayer._tcp.local.`。
2. TXT 记录包含 `uuid`、`alias`、`mode`、`androidVersion`、`rtpPort`、`controlPort`。
3. Windows 和 Android 配置端可额外发送 UDP 广播 `DISCOVER_SOUNDPLAYER`，声音端返回 JSON 设备信息。
4. DLNA 模式同时发布 SSDP/UPnP DMR 服务。

### 6.3 控制 API
基础地址：`http://<device-ip>:5006/api`

| 接口 | 方法 | 说明 |
|------|------|------|
| `/device` | GET | 获取设备信息 |
| `/config` | GET/POST | 获取或设置模式、声道、别名、延迟 |
| `/status` | GET | 获取信号、缓冲、丢包、播放状态 |
| `/calibration/start` | POST | 开始延迟/声道同步测试 |
| `/dlna/group` | POST | 设置 DLNA 多机组、声道和起播参数 |

### 6.4 RTP 包要求
1. RTP Header 保持标准 12 字节，`PayloadType=96` 表示 Opus。
2. 初版不强制使用 RTP extension，声道和格式可以通过控制通道下发，降低 Android 4.4 解析复杂度。
3. 若后续启用 RTP extension，Android 解析器必须正确跳过扩展头，不能把扩展头当音频数据。
4. RTP timestamp 使用采样计数，不使用任意微秒时间戳，便于音频播放调度。

## 7. 低延迟和音质优化
### 7.1 声卡模式
1. Opus 使用 48kHz，10ms 或 20ms 帧长；低延迟优先 10ms，高质量优先 20ms。
2. 码率建议：单声道 64-96kbps，立体声 128-192kbps，高质量 256kbps。
3. 开启 Opus PLC，网络差时开启 in-band FEC。
4. 自适应抖动缓冲：低延迟 40-80ms，稳定优先 80-150ms。
5. Android 端避免频繁 GC，音频线程使用固定缓冲池。

### 7.2 DLNA 模式
1. DLNA 不作为低延迟路径，优先稳定播放和兼容性。
2. 多机 DLNA 需要统一预缓冲，起播前每台设备进入 `READY` 后再发 `PLAY_AT` 指令。
3. 每台设备独立测量解码启动延迟，保存补偿值。
4. 对 Android 4.4 老设备，DLNA 多机偏差可能明显，应提供一键切换声卡模式。

## 8. 开发路线图
| 阶段 | 时间 | 核心任务 | 交付物 |
|------|------|----------|--------|
| Phase 1 | 2-4 周 | 声音端固定标识、mDNS/UDP 扫描、WASAPI 捕获、RTP 播放 | 可扫描、可命名、可左/右声道播放 |
| Phase 2 | 4-6 周 | 声卡模式低延迟优化、2.1/5.1/7.1 声道、高级配置 UI | 多设备声道系统可用 |
| Phase 3 | 4-8 周 | DLNA DMR、DLNA 单机/多机组、延迟校准 | DLNA 兼容和多机配置可用 |
| Phase 4 | 8-12 周 | Windows 虚拟声卡驱动、安装器、驱动签名、系统默认输出 | 真正系统级声卡输出 |
| Phase 5 | 持续 | 稳定性、兼容机型、路由器环境测试 | 可发布版本 |

## 9. 主要风险与处理
| 风险 | 影响 | 处理 |
|------|------|------|
| DLNA 多机天然不同步 | 左右声道错位 | 保留功能但强提示校准；推荐声卡模式做严格同步 |
| Android 4.4 音频延迟不可控 | 低延迟体验不稳定 | 兼容播放优先，按设备分级降级 |
| 虚拟声卡驱动复杂 | 产品化周期长 | WASAPI 先验证，驱动独立阶段开发 |
| 多网卡扫描失败 | 找不到设备 | mDNS + UDP 广播 + 手动网卡绑定 |
| WiFi 抖动和丢包 | 爆音/延迟波动 | 自适应缓冲、Opus PLC/FEC、码率动态调整 |
| 文档与代码协议不一致 | 三端无法互通 | 以 5004/5005/5006 + mDNS 为当前基线，修改协议需三端同步 |

## 10. 结论
本方案按你的需求保留三类能力：Android 4.4+ 声音端、Windows 系统级声卡发送端、Android 配置端，以及 DLNA 单机/多机模式。真正追求低延迟、音质和多设备同步时，应以电脑声卡模式作为主链路；DLNA 多机模式作为兼容增强功能实现，必须依赖同源音轨、统一起播、声道拆分和延迟补偿，不能按普通 DLNA 投屏方式直接承诺左右声道无延迟。
